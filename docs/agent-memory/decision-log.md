# 决策日志（Decision Log）

> 仅记录会影响后续实现方式、验证方式或回滚成本的决策。

## [2026-02-20] 决策：设备新增入口收敛到画布左上纵向工具栏，设备绑定切换下沉到卡片内
- 背景：左侧独立“新增设备面板”与画布操作分离，导致新增后经常出现在非关注区域，且设备选择路径较长。
- 备选方案：A. 继续保留左侧新增面板 / B. 迁移新增入口到画布工具栏并在卡片内直接切换绑定 / C. 新增弹窗完成创建与绑定
- 结论：采用 B。
- 原因：与画布操作路径一致，新增后可立即可见（视图中心落点），减少“先选设备再创建”的跳转成本。
- 影响范围：`src/App.tsx`（移除左侧创建区与创建参数）、`src/components/RoutingCanvas.tsx/.css`（左上纵向工具栏、视图中心落点）、`src/components/DeviceCard.tsx/.css`（卡片内绑定下拉与回采配色）。
- 回滚策略：可恢复左侧创建面板组件并保留当前卡片内绑定切换能力，互不冲突。

## [2026-02-20] 决策：先采用线性重采样 + 目标延迟自适应纠偏
- 背景：系统输入/输出设备采样率不一致会导致时钟漂移、欠载或爆音。
- 备选方案：A. 严格要求同采样率 / B. 线性重采样 + 缓冲纠偏 / C. 直接引入高阶重采样库
- 结论：采用 B 作为当前阶段默认策略。
- 原因：实现复杂度可控、实时性好，能够快速显著提升稳定性与可用性。
- 影响范围：`src-tauri/src/lib.rs` 的 `StreamBridgeState`、输入入队与输出取样逻辑。
- 回滚策略：若需更高音质，可在保持缓冲框架不变的前提下替换插值算法。

## [2026-02-20] 决策：系统级音频引擎采用“工作线程 + 命令通道”
- 背景：`cpal::Stream` 不是 `Send`，不能直接放在 Tauri `State`（要求 `Send + Sync`）中。
- 备选方案：A. 将流对象存入全局状态 / B. 工作线程持有流对象，主线程通过通道发命令 / C. 放弃系统级，继续前端音频
- 结论：采用 B。
- 原因：满足 Tauri 线程安全约束，同时保留系统级音频执行能力，结构清晰可扩展。
- 影响范围：`src-tauri/src/lib.rs` 新增音频引擎线程、命令协议与状态查询命令。
- 回滚策略：可回退到仅前端 WebAudio 路径（保留 UI，移除后端音频命令调用）。

## [2026-02-20] 决策：真实音频能力先走前端 WebAudio 实时图
- 背景：需要尽快实现“真实电平 + 虚拟设备可串流”，但当前 Tauri 后端无音频流命令和依赖。
- 备选方案：A. 先实现前端 WebAudio 图 / B. 立即扩展 Rust 后端做系统级音频引擎 / C. 继续模拟电平
- 结论：先采用 A，落地应用内真实采集与串流；系统级虚拟声卡能力作为下一阶段。
- 原因：A 改动面最小、验证最快、可直接满足“真实电平与串流”核心体验；B 成本高且需要额外驱动/平台适配。
- 影响范围：`src/hooks/useAudioRouter.ts` 的音频图与设备语义；`isVirtual` 当前解释为“应用内虚拟总线”。
- 回滚策略：若后续改为后端音频引擎，可保留前端设备/连线 UI，仅替换 hook 内部实现与命令调用层。
