# Core Link · AGENTS 协作规范

本文件定义 AI Agent 与人类协作者的统一工作方式，重点解决 **Agent 记忆丢失（上下文丢失、会话中断、模型切换）** 问题。

---

## 1. 目标与原则

### 1.1 目标
- 保证任意 Agent 在中断后可快速恢复到可执行状态。
- 降低“重复分析、重复改动、遗漏约束”的概率。
- 让需求、决策、实现、验证可追溯。

### 1.2 协作原则
- 单一事实来源（Single Source of Truth）：以仓库内文档为准，不依赖聊天历史。
- 任务启动强制阅读：进入 Plan 前必须阅读 `DESIGN.md`，确保实现与产品范围一致。
- 小步提交上下文：每完成一个阶段都写快照。
- 决策显式化：关键取舍必须记录“为什么”。
- 可恢复优先：任何时刻都能被新 Agent 接手。

---

## 2. Agent 角色定义

### 2.1 Planner（规划）
- 拆解任务、识别依赖、定义验收标准。
- 产出：`任务计划`、`风险清单`、`完成定义`。

### 2.2 Builder（实现）
- 仅按计划范围改动代码。
- 产出：`变更摘要`、`文件清单`、`回滚点`。

### 2.3 Verifier（验证）
- 执行构建、测试、关键路径手测。
- 产出：`验证结果`、`失败复现`、`建议修复`。

### 2.4 Recorder（记录）
- 持续更新会话记忆工件（见第 4 节）。
- 产出：`会话快照`、`决策日志`、`交接摘要`。

> 小任务可由一个 Agent 兼任多角色，但职责不能省略。

---

## 3. 标准任务状态机

每个任务必须按以下状态流转：

`Intake -> Plan -> Implement -> Verify -> Handoff/Done`

每个状态结束时都要写入最小记忆快照（见第 4 节），避免中断后丢失上下文。

---

## 4. 防记忆丢失工作流（核心）

## 4.1 记忆分层

### L0：即时上下文（易丢失）
- 聊天窗口、临时思路、未落盘分析。
- 规则：**不可信，不可作为唯一依据**。

### L1：会话快照（高频落盘）
- 当前目标、进行中步骤、下一步动作、阻塞项。
- 更新频率：每完成一个阶段或每 15-20 分钟。

### L2：项目记忆（长期）
- 稳定约束、架构决策、术语定义、关键命令。
- 仅在事实变化时更新。

### L3：交付证据（可审计）
- 改动文件、测试结果、风险说明、回滚方式。
- 在合并前必须完整。

---

## 4.2 目录与工件约定

建议在仓库中维护以下记忆文件：

- `docs/agent-memory/project-context.md`（L2）
- `docs/agent-memory/decision-log.md`（L2）
- `docs/agent-memory/session-log.md`（L1）
- `docs/agent-memory/handoff.md`（L1/L3）

当前仓库已落地：`docs/agent-memory/`。

执行要求：
- 新任务开始后，优先更新 `session-log.md` 当前条目。
- 任务完成前，必须更新 `handoff.md`。
- 存在关键取舍时，必须补充 `decision-log.md`。

最小要求：如果暂未创建目录，至少在当前 `AGENTS.md` 中遵守第 4.4 的模板字段。

---

## 4.3 会话恢复 SOP（Standard Operating Procedure）

当出现模型切换、窗口关闭、上下文截断时，新 Agent 必须按以下顺序恢复：

1. 读取 `AGENTS.md`、`DESIGN.md` 与最近一次 `handoff`。
2. 提取 `任务目标 / 当前状态 / 未完成步骤 / 风险`。
3. 对照代码现状确认“计划是否过时”。
4. 输出 5 行以内恢复结论：
	- 我理解的目标
	- 我将立即执行的第一步
	- 我认为最大的风险
	- 预期验证方式
	- 若信息不足需要的最小补充
5. 开始执行，并在首个里程碑后补写 `session-log`。

目标：**1 分钟内恢复可执行状态，5 分钟内产出首个有效改动或验证结果。**

---

## 4.4 强制模板（复制即用）

### A. Session Log 条目模板

```md
## [YYYY-MM-DD HH:mm] 会话快照
- 任务：
- 当前状态：Intake | Plan | Implement | Verify | Handoff
- 已完成：
- 进行中：
- 下一步（可直接执行）：
- 阻塞项：
- 涉及文件：
```

### B. Decision Log 条目模板

```md
## [YYYY-MM-DD] 决策：<标题>
- 背景：
- 备选方案：A / B / C
- 结论：
- 原因：
- 影响范围：
- 回滚策略：
```

### C. Handoff 模板

```md
## 交接摘要
- 目标：
- 当前进度：
- 剩余任务（按优先级）：
- 风险与注意事项：
- 验证现状（已通过/未通过）：
- 建议下一条指令：
```

## 5. 执行规范（面向 Core Link）

### 5.1 UI/交互
- 保持专业音频工具风格：信息密度高但结构清晰。
- 新功能优先遵循现有组件与主题，不私增视觉体系。

### 5.2 代码改动
- 优先修复根因，不做表面补丁。
- 单次任务只改与目标直接相关的内容。
- 任何接口行为变化必须在交接中明确。

### 5.3 验证
- 先跑最小相关验证，再扩大验证范围。
- 若存在未修复失败项，需标明“是否与本次改动相关”。

---

## 6. 完成定义（Definition of Done）

满足以下条件才可标记 Done：

- 需求已实现并与约束一致。
- 关键路径验证完成，结果可复现。
- 已写入最新 `session-log` 与 `handoff`。
- 关键决策已写入 `decision-log`（如有）。
- 下一位接手者可在 1 分钟内理解并继续。

---

## 7. 快速检查清单（每次任务结束前）

- 我是否把“下一步动作”写成可直接执行的一句话？
- 我是否记录了为什么这样做，而不只是做了什么？
- 我是否标注了风险与回滚点？
- 我是否让新 Agent 不看聊天记录也能接手？

如果以上任一为“否”，说明记忆仍有丢失风险，不能结束任务。
